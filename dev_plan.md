# 分流器与解析引擎开发流程 — 详细规范

## 一、核心原则

| 原则 | 说明 |
|:---|:---|
| **分类优先** | 脚本进来，先分类，再解析。分类决定走哪条管线，解析负责产出步骤。 |
| **管线隔离** | 每条管线方法自包含。不修改共享函数，不修改其他管线的去重/合并逻辑。 |
| **不破坏基线** | ZS 目录中已验收的脚本是基线。任何改动不得使基线脚本的步骤内容变化。 |
| **AI 验收不调引擎** | AI 逐步骤对比"阅读脚本理解出的正确流程"与"解析结果输出"。步数一致不算验收通过，必须每步的 type/part/fileName 全部对得上。 |

## 二、核心对象

| 对象 | 职责 | 所在文件 |
|:---|:---|:---|
| **分类器** | 分析脚本结构特征 → 输出 class_id | `classifier.py` |
| **解析管线映射** | class_id → 管线方法 | `__init__.py` 第92-111行 |
| **管线方法** | 根据 class_id 解析出步骤列表 | `__init__.py` 各 `_parse_xxx` 方法 |
| **ZS 目录** | 已验收脚本基线库，按 `ZS/{type}/{class_id}/` 组织 | `ZS/` |

## 三、开发流程（正向：新脚本接入）

```
拿到新脚本
  ↓
Step 1 — AI 阅读脚本，写出逐步骤预期流程
  不需要跑引擎，直接读脚本源码。
  逐行理解脚本逻辑，写出的预期流程包含每步的类型、分区名、文件名。
  条件分支走"所有条件都满足"的最完整路径。
  通配符 for 循环标注为通配符步骤（不是"不知道"）。
  ↓
Step 2 — AI 跑一次分类器+管线解析，获得解析结果
  写法：写一个临时文件 _test_one.py，一次性输出：
    class_id + 总步数 + 每一步的 type/part/fileName/raw
  跑一次终端命令，拿到全部输出文本。
  ↓
Step 3 — AI 逐步骤对比
  用 Step 1 的预期流程与 Step 2 的输出逐行对账。
  ├─ 完全一致 → ✅ 验收通过，移入 ZS/{type}/{class_id}/
  ├─ 部分不一致 → 进入 §四
  └─ 0 步或明显错误 → 直接进入 §四「解析能力不足」
```

## 四、不一致处理

```
对比结果不一致
  ↓
判断根因：

  ① 分类粒度问题
    新脚本与 ZS 中同 class_id 的已有脚本结构差异大，class_id 太粗。
    → 拆分子类 + 新增管线方法。不影响已有脚本。
    → 在 classifier.py 中增加判定规则，区分新旧脚本。

  ② 解析能力不足
    分类正确（class_id 对），但管线实现不完整。
    → 进入「增强流程」（§五）
    
  ③ 脚本本身有歧义
    条件分支在解析器无法判断时产生占位步骤。
    这是可接受的（标注 confidence=estimated/placeholder 即可）。
    → 验收通过。
```

## 五、增强流程（⚠️ 含回测 + 回退规则）

**触发条件**：分类正确但管线实现不完善。

```
Step E1 — 定位管线内缺失的逻辑
  在管线方法内部修复。
  禁止修改共享函数。

Step E2 — 用新脚本验证
  增强后，新脚本的解析结果与 AI 理解一致。

Step E3 — 回测同管线基线脚本
  对 ZS/{type}/{class_id}/ 下的所有已有脚本重新跑解析。
  AI 逐步骤对比：
  ├─ 步骤无变化 → ✅ 增强通过，验收新脚本
  └─ 已有脚本步骤出现退化（步骤内容变了/步数增减）
      → ❌ 放弃增强，回滚代码
      → 进入「重新分类」（§六）
```

**如果增强导致基线退化，必须重新分类，不在原管线上继续修。**

## 六、重新分类（增强失败的回退）

1. **回滚所有修改**
2. 分析新脚本与已有脚本的**结构性差异**
3. 在 `classifier.py` 增加判定规则，将新脚本分出新的 class_id
4. 在 `pipeline_map` 中添加新管线方法
5. 实现新管线方法
6. 验收新脚本 → 移入 ZS

**原管线和已有基线脚本完全不受影响。**

## 七、验收标准

| 通过条件 | 不通过条件 |
|:---|:---|
| 每步 type/part/fileName 全部匹配 AI 理解 | 步数对但步骤内容错（假阳性） |
| 通配符步骤标注为 placeholder | 0 步输出（空解析） |
| 条件分支默认走完整路径 | 函数体内命令泄漏为独立步骤 |

## 八、禁止事项

| 禁止 | 原因 | 正确替代 |
|:---|:---|:---|
| 修改共享函数 | 影响所有管线 | 写在各自管线方法内部 |
| classifier 用文件名匹配 | 分类应基于内容结构 | 用正则匹配脚本结构特征 |
| 增强后不回测 | 可能破坏基线 | 回测 ZS 同管线目录 |
| 回测失败后继续修 | 管线变脆弱 | 回滚 → 重新分类 |
| 管线内为特定脚本加 if/else | 管线混乱 | 增强全管线覆盖 或 拆类 |
| 用步数对比代替逐步骤对账 | 步数对 ≠ 内容对 | AI 必须逐步骤对比 |
| 反复写 _test_one.py 调引擎来验证 | 效率低、链路长、易出错 | 一次终端运行输出全部结果，AI 直接对比 |

## 十、样本标注闭环（天树引擎浇水）


### 命名规则

通过 WebDAV 操作 OpenList 上的**文件名前缀**来标注天树引擎支持状态。

| 前缀 | 图标 | 含义 | 条件 |
|:---|:---|:---|:---|
| `yes_` | ✅ | 天树引擎支持 | `total_steps > 0`（无论有无警告） |
| `no_` | ❌ | 天树引擎不支持 | `total_steps == 0` |
| （无前缀） | ❓ | 待判定 | 尚未被引擎处理 |


### 判定粒度

- 判定依据是**调用天树引擎 parse() 的真实结果**（`total_steps`），不使用元数据快照
- 引擎升级后重新判定可能翻转（no→yes），属正常行为
- 不重复标记：文件名已是 `yes_` 或 `no_` 开头时跳过
- WebDAV 上只有 `.bat/.cmd/.sh/.txt` 文件参与标注


### 数据流

```
OpenList(WebDAV)                    服务器                       前端
    │                                 │                           │
    │ ① PROPFIND 列目录               │                           │
    │ ←────────────────────────────── │                           │
    │                                 │ ② 筛选无前缀的待判定文件   │
    │                                 │                           │
    │ ③ GET 下载脚本内容              │                           │
    │ ←────────────────────────────── │                           │
    │                                 │ ④ 天树引擎 parse()        │
    │                                 │    total_steps > 0 → yes_ │
    │                                 │    total_steps == 0 → no_ │
    │                                 │                           │
    │ ⑤ MOVE 重命名（加 yes_/no_）    │                           │
    │ ←────────────────────────────── │                           │
    │                                 │ ⑥ list API               │
    │                                 │ ────────────────────────→ │
    │                                 │                  按前缀   │
    │                                 │                  渲染图标 │
```


### 操作流程

1. **拉取文件列表** — WebDAV PROPFIND 获取远程全部文件名
2. **筛选待判定** — 过滤出无 `yes_`/`no_` 前缀的脚本文件
3. **下载内容** — WebDAV GET 下载到本地临时目录
4. **天树引擎解析** — 调用 `HydraEngine.parse()` 逐文件判定
5. **计算前缀** — `total_steps > 0` → `yes_`，`total_steps == 0` → `no_`
6. **反写文件名** — WebDAV MOVE 重命名（加前缀）
7. **前端展示** — 列表 API 根据文件名前缀渲染 ✅/❌/❓


### 引擎升级后的重新判定

1. 手动触发 `/api/script/relabel` 接口
2. 下载 WebDAV 上所有 `no_` 前缀的文件
3. 用最新引擎重新 parse()
4. 如 now_supported → MOVE 改名 `yes_`（翻转）
5. 如 still_unsupported → 保持 `no_`
6. 操作日志记录翻转明细到 `UPLOAD_DIR/label_log.json`


### 前端展示

```
yes_xxx.bat  →  ✅ 可解析
no_xxx.bat   →  ❌ 待适配
xxx.bat      →  ❓ 待判定
```

状态图标在文件名左侧，醒目可见。点击仍可查看脚本内容。


### 安全与日志

- WebDAV 认证复用 `config.py` 中现有凭据
- MOVE 操作前检查目标不存在，防止覆盖
- 每次标注操作记录：时间、原文件名、新文件名、步数、判定结果
- 操作日志保存在 `UPLOAD_DIR/label_log.json`
