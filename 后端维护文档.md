## 1. 项目总览

天树刷机是一个基于 Flask 的 Android Fastboot 线刷工具，部署在手机本地（Termux），通过 `127.0.0.1:8080` 提供网页界面。

### 技术栈

| 组件 | 技术 |
|---|---|
| 后端 | Python 3 + Flask |
| 前端 | 原生 HTML/CSS/JS（无框架） |
| 通信 | HTTP REST API + WebSocket (Socket.IO) |
| 刷机工具 | fastboot (AArch64 二进制) |
| 配置格式 | YAML（可选，无 PyYAML 时用内置默认值） |

### 核心设计原则

- **三层加载模型**：包内代码（静态） → runtime/ 目录（热更新） → VIP 逃生通道（文件名映射）
- **特征向量分类**：12 BAT + 15 SH 正交特征，class_id 由命中特征拼接生成
- **配置自动生效**：所有 runtime/ 下的配置文件修改后自动检测 mtime 变更，无需重启
- **服务器同步**：服务器端管理配置，本地端通过 API 拉取同步

## 3. 核心架构：天树引擎 (Hydra)

天树引擎是脚本解析的核心，位于 `core/hydra/__init__.py`。

### 解析流程

```
用户上传脚本
    │
    ▼
rom_handler._read_and_parse_script()
    │
    ▼
HydraEngine.parse(content, script_type, rom_dir, script_path)
    │
    ├─ 1. VIP 逃生通道检查
    │     vip_loader.match_vip(script_name)
    │     命中 → 调用 VIP parse()，直接返回
    │
    ├─ 2. 特征向量分类
    │     classifier.classify(content, script_type)
    │     → 返回 class_id（如 "bat:for+nested+if+delayed"）
    │
    ├─ 3. 管线注册表查询
    │     registry.get_pipeline(class_id)
    │     4 级查询：精确覆写 → 前缀覆写 → 包路由 → 基类
    │
    ├─ 4. 管线执行
    │     pipeline.parse(content, script_path, rom_dir)
    │     BAT: bat_sandbox 模拟执行
    │     SH:  sh_parser 模拟执行
    │
    └─ 5. 返回 HydraParseResult
          steps[] + blocks[] + missing_files[]
```

### 关键数据结构

```python
# core/hydra/__init__.py
@dataclass
class HydraStepCompat:
    type: str          # flash/erase/reboot/set_active/oem/adb
    part: str          # 分区名
    fileName: str      # 镜像文件名/路径
    params: str        # 额外参数
    raw: str           # 原始命令行
    risk: str          # HIGH/MEDIUM/LOW
    dynamic: bool      # 是否动态步骤
    loop: dict = None  # 循环信息
    call: dict = None  # 调用信息
    condition: str = None  # 条件

@dataclass
class HydraParseResult:
    steps: List[HydraStepCompat]
    total_steps: int
    missing_files: List[str]
    blocks: List[CodeBlock]
    script_type: str   # bat/sh
    class_id: str      # 分类ID
```

## 4. 脚本分类系统

### 分类器位置

`core/hydra/classifier.py` — 全局单例 + mtime 缓存

### 工作原理

分类器对脚本内容做多个**独立的正则特征检测**，每个特征命中为 `true` 或 `false`。class_id 由所有 `true` 的特征 short 名按定义顺序用 `+` 拼接生成。

```
脚本内容 → [特征1: for?] [特征2: if?] [特征3: delayed?] ... → class_id
```

### 示例

| 脚本特征 | class_id |
|---|---|
| 纯 fastboot 命令 | `bat:bare` |
| 有 for 循环 + if 条件 | `bat:for+if` |
| 交互式 + for_f + goto + if + delayed + dyn | `bat:interactive+for_f+goto+if+delayed+dyn` |
| SH 脚本有 if + test + pipe | `sh:if+test+pipe` |

### 配置文件

`runtime/rules/classifier.yaml`：

```yaml
bat_features:
  - id: interactive
    short: interactive
    pattern: '\bset\s+/p\b'
    flags: I                  # I=不区分大小写, S=跨行, M=多行

  - id: for_loop
    short: for
    pattern: '\bfor\s+%%\w+\s+in\s*\(\s*[^*]'
    flags: I

  - id: delayed_expansion
    short: delayed
    pattern: '!\w+!'         # 延迟展开变量 !var!
    flags: I
    requires_also: 'enabledelayedexpansion'  # 附加条件

sh_features:
  - id: for_loop
    short: for
    pattern: '\bfor\s+\S+\s+in\b'
    flags: I
```

### 特征字段说明

| 字段 | 必填 | 说明 |
|---|---|---|
| `id` | 是 | 唯一标识 |
| `short` | 是 | class_id 中的缩写 |
| `pattern` | 是 | 正则表达式 |
| `flags` | 是 | `I`/`S`/`M`，可组合如 `IS` |
| `is_fallback` | 否 | `true` 表示仅当无其他特征命中时才激活 |
| `marker` | 否 | `true` 表示不参与 class_id，仅作为附加标记 |
| `requires_also` | 否 | 附加条件正则，两者都命中才算 true |

### mtime 缓存

分类器是全局单例（`ScriptClassifier._instance`），每次 `classify()` 调用时检查 `classifier.yaml` 的 mtime，文件未修改则用缓存的规则列表，修改了才重新读取 YAML。

## 5. 管线注册表 (Pipeline Registry)

### 位置

`core/hydra/pipelines/registry.py`

### 4 级查询

当分类器返回 class_id 后，注册表按以下优先级查找管线：

```
1. 精确覆写    runtime/classes/{class_id}/pipeline.py
              ↑ 完全匹配 class_id

2. 前缀覆写    runtime/classes/{prefix}/pipeline.py
              ↑ class_id 的前缀匹配（如 bat:for+if 匹配 bat:for）

3. 包路由      pipelines/bat/interactive/pipeline.py
              ↑ 交互式脚本走专用管线
              ↑ 判断条件：class_id 包含 "interactive"

4. 基类        pipelines/bat/pipeline.py (BatPipeline)
              pipelines/sh/pipeline.py  (ShPipeline)
              ↑ 兜底，所有未匹配的 class_id 走基类
```

### 管线基类

```python
# pipelines/bat/pipeline.py
class BatPipeline:
    def __init__(self, class_id="bat:bare"):
        self.class_id = class_id
        self.sandbox = None  # 延迟初始化

    def parse(self, content, script_path="", rom_dir=""):
        # 1. 创建 BAT 沙箱
        # 2. 加载脚本内容
        # 3. 模拟执行
        # 4. 收集 fastboot 步骤
        # 5. 返回 HydraParseResult
```

### 子类覆写

在 `runtime/classes/` 下创建以 class_id 命名的目录：

```
runtime/classes/bat:for+goto+if/pipeline.py
```

```python
from core.hydra.pipelines.bat.pipeline import BatPipeline

class Pipeline(BatPipeline):
    class_id = "bat:for+goto+if"

    def parse(self, content, script_path="", rom_dir=""):
        result = super().parse(content, script_path=script_path, rom_dir=rom_dir)
        # 自定义修改 result.steps
        return result
```

### 加载器

`pipelines/loader.py` 使用 `importlib.util.spec_from_file_location` 动态加载 `runtime/classes/` 下的 `pipeline.py`，带 mtime 缓存——文件修改后自动重新加载。

## 6. VIP 逃生通道

### 位置

`core/hydra/vip_loader.py`

### 用途

当标准管线无法正确解析某个脚本时，VIP 通道允许完全绕过 sandbox，用自定义 Python 代码解析。

### 配置文件

`runtime/vip/registry.yaml`：

```yaml
# 格式："脚本文件名": VIP解析器名
"抉择线路.bat": jueline
"D1bat.bat": d1bat

# 支持通配符
"wildcard_*.bat": my_parser
```

### VIP 解析器

`runtime/vip/my_parser.py`：

```python
def parse(content, script_path="", rom_dir=""):
    from core.hydra import HydraParseResult, HydraStepCompat
    from core.hydra.bat_parser.var_types import CodeBlock, HydraStep

    steps = []
    for line in content.splitlines():
        line = line.strip()
        if line.startswith('fastboot flash'):
            parts = line.split()
            steps.append(HydraStepCompat(
                type='flash', part=parts[2], fileName=parts[3],
                params='', raw=line, risk='MEDIUM'
            ))

    bs = [HydraStep(command=f"{s.type} {s.part}", subcommand=s.type,
                    partition=s.part, path=s.fileName, risk=s.risk) for s in steps]
    block = CodeBlock("vip", bs, "VIP解析", "MEDIUM")
    return HydraParseResult(steps=steps, total_steps=len(steps),
                           missing_files=[], blocks=[block],
                           script_type="bat", class_id="vip:my_parser")
```

### 自动刷新

`vip_loader.match_vip()` 每次调用时检查 `registry.yaml` 的 mtime，文件修改后自动重新加载映射表。VIP `.py` 文件也通过 mtime 缓存自动检测变更。

### 交互式选择项

`runtime/vip/choices.yaml`：

```yaml
# 顺序抉择：针对特定脚本的分步选择
```

## 7. 热更新框架

### 概述

所有 `runtime/` 目录下的配置文件修改后**自动生效**，无需重启服务。系统在每次解析脚本时通过 mtime 检测文件变更。

### 热更新范围

| 配置文件 | 自动检测机制 | 缓存位置 |
|---|---|---|
| `runtime/rules/classifier.yaml` | ScriptClassifier 单例检查 mtime | `ScriptClassifier._instance` |
| `runtime/vip/choices.yaml` | `_load_choices()` 检查 mtime | `_choices_cache` |
| `runtime/vip/registry.yaml` | `match_vip()` 检查 mtime | `VipLoader._registry_mtime` |
| `runtime/vip/*.py` | `load_parser()` 检查 mtime | `VipLoader._cache` |
| `runtime/classes/*/pipeline.py` | `load_class()` 检查 mtime | `PipelineLoader._cache` |

### 手动刷新

网页「版本」页顶部「⚡ 配置热更新」区域：
- **🔄 刷新配置** — 调用 `hot_reload.reload_all()`，清空所有缓存
- **📊 查看状态** — 调用 `hot_reload.get_status()`，显示加载的配置数量

### API 端点

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/api/hydra/reload` | 全量重载 |
| POST | `/api/hydra/reload-rules` | 仅重载分类规则 |
| GET | `/api/hydra/status` | 查看加载状态 |

### 相关文件

| 文件 | 职责 |
|---|---|
| `core/hydra/hot_reload.py` | 热更新入口函数 |
| `routes/api_hydra.py` | 热更新 API 路由 |
| `core/hydra/classifier.py` | 分类器（单例 + mtime） |
| `core/hydra/vip_loader.py` | VIP 加载器（mtime） |
| `core/hydra/pipelines/loader.py` | 管线加载器（mtime） |

## 9. ROM 解析流程

### 入口

`core/rom_handler.py` → `_read_and_parse_script(filepath)`

### 流程

```
1. 读取脚本文件（尝试 GBK 解码，失败用 UTF-8）
2. 推断 rom_dir（脚本所在目录）
3. 调用 HydraEngine.parse() 解析
4. _hydra_steps_to_old() — 转换为旧版 step dict
5. _resolve_image_paths() — 解析镜像绝对路径
6. _expand_wildcard_steps() — 展开通配符 FOR 循环
7. optimize_step_order() — 步骤排序（保持原始顺序，仅标记 COW 清理）
```

### 镜像路径解析

`_resolve_image_paths(steps, rom_dir)` 的逻辑：

```
对每个 flash 步骤的 fileName：
  1. 标准化路径分隔符（\ → /），去除引号
  2. 如果是绝对路径（sandbox 已展开 %~dp0）→ 直接使用
  3. 如果是相对路径 → 拼接 rom_dir
  4. 检查文件是否存在 → 存在则写入 step["imagePath"]
```

### 前后端一致性

| 维度 | 一致性 | 说明 |
|---|---|---|
| 步骤顺序 | 一致 | `optimize_step_order` 不重排序 |
| 镜像路径 | 一致 | 前端用 `imagePath`，后端优先用 `imagePath`，回退到 `get_image_path()` |
| 命令参数 | 一致 | `prefixParams` 在 `flash` 之前（如 `fastboot --disable-verity flash vbmeta ...`） |

## 10. 刷机执行流程

### 单分区刷机

```
api_flash.py → flasher.flash_partition()
  → get_image_path(source, image_name, rom_name)
      source="custom" → 直接使用绝对路径
      source="rom"    → 拼接 ROM 目录
      source="local"  → 从镜像目录查找
  → create_flash_task(partition, image_path, extra, allow_dangerous)
  → 执行 fastboot flash {partition} {image_path} {extra}
```

### 批量刷机

```
api_batch.py → batch_flasher.batch_flash_worker()
  → batch_precheck() — 预校验（文件存在性、高危分区、电量）
  → 逐步执行：
      flash 步骤 → 优先用 step["imagePath"]，回退 get_image_path()
      erase 步骤 → fastboot erase {part}
      reboot 步骤 → fastboot reboot
  → WebSocket 实时推送进度
```

### 镜像来源类型

| source | 说明 | 路径处理 |
|---|---|---|
| `file` | 用户直接选择文件上传 | 上传到 IMAGE_DIR，后端用 `custom` 源处理 |
| `custom` | 自定义绝对路径 | `get_image_path()` 直接使用绝对路径 |
| `rom` | 线刷包镜像 | 拼接 ROM 目录 |
| `local` | 手机目录镜像 | 从 IMAGE_DIR 查找 |

## 12. 安全机制

### 路径遍历防护

`core/utils.py` → `sanitize_path(base_dir, user_path)`：
- URL 解码后规范化路径
- 解析符号链接
- 确保结果路径在 base_dir 内

### 危险命令黑名单

`core/utils.py` → `is_dangerous_command(cmd)`：

```python
_DANGEROUS_PATTERNS = [
    r'rm\s+(-[rf]+\s+)+/',      # rm -rf /
    r'rm\s+(-[rf]+\s+)+~',      # rm -rf ~
    r'mkfs\.',                    # 格式化
    r'dd\s+.*of=/dev/',          # dd 写入设备
    r'dd\s+if=/dev/zero',
    r'shutdown',
    r'>\s*/dev/sd',
    r'flash_\s*all',
    r':\(\)\s*\{\s*:\|:\s*&\s*\}\s*;',  # fork bomb
]
```

### 配置同步安全白名单

- 只允许 `rules/`、`vip/`、`classes/` 目录
- 只允许 `.yaml`、`.yml`、`.py` 扩展名
- 拒绝包含 `..` 的路径

### 分区名校校验

`validate_partition_name()` — 只允许字母、数字、下划线。

### 镜像文件名校校验

`validate_image_filename()` — 检查 `..`、`\`、`\x00` 等危险字符。
`custom` 源跳过扩展名校验（支持任意格式文件）。

## 15. 常见问题排查

### Q: 上传脚本后解析结果为空

**排查步骤**：
1. 查看日志 `flash_tool.log`，搜索「天树引擎解析为空」
2. 检查脚本是否包含 `fastboot` 命令
3. 检查 `classifier.yaml` 是否正确加载（网页「版本」→「📊 查看状态」）
4. 如果脚本是特殊格式，考虑添加 VIP 解析器

### Q: 镜像路径不正确

**排查步骤**：
1. 检查 `rom_dir` 是否正确（脚本所在目录）
2. 检查 `_resolve_image_paths()` 是否正确处理了 `%~dp0` 变量
3. 查看 `imagePath` 字段是否为绝对路径
4. 如果是 `custom` 源，检查路径是否存在

### Q: 交互式脚本选择项不显示

**排查步骤**：
1. 检查 class_id 是否包含 `interactive`
2. 检查 `choices.yaml` 是否定义了该脚本
3. 检查 PyYAML 是否安装
4. 检查上传的文件名是否能匹配 `sequential_choices` 中的 key（支持子串匹配）

### Q: 热更新不生效

**排查步骤**：
1. 确认修改的文件在 `runtime/` 目录下
2. 点击网页「🔄 刷新配置」按钮
3. 检查文件权限（确保 Flask 进程可读）
4. 查看 `flash_tool.log` 是否有加载错误

### Q: 服务器同步失败

**排查步骤**：
1. 检查服务器地址是否正确
2. 检查网络连接（手机能否访问服务器）
3. 检查服务器端 Flask 是否正常运行
4. 查看日志中的 `URLError` 信息

### Q: vbmeta 参数 `--disable-verity` 丢失

**排查步骤**：
1. 检查 `executor.py` 的 `_intercept_fastboot()` 是否正确收集了 `global_params`
2. 检查 `_hydra_steps_to_old()` 是否正确提取了 `prefixParams`
3. 检查 `generate_preview_commands()` 是否将 `prefixParams` 放在 `flash` 之前
